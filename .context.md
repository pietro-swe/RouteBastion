# RouteBastion Project Context

## Overview

RouteBastion is a monorepo for a route optimization broker platform focused on Vehicle Routing Problem (VRP) integrations. Its main purpose is to provide a unified interface between external client systems and multiple route optimization providers, reducing provider lock-in and hiding provider-specific API shapes, constraints, and operational differences.

The project currently contains three main applications:

- `apps/broker`: the core broker API, implemented in Go.
- `apps/admin-api`: an administration API, implemented with NestJS.
- `apps/admin-ui`: an administration UI, implemented with Vue 3 and Vite.
- `apps/packages`: Shared packages (such as contracts) for applications within the monorepo.

The repository also includes PostgreSQL schemas and migrations, Docker Compose development infrastructure, Kong gateway configuration, load testing scripts, and Kubernetes manifests for broker deployment.

## Product Goals

RouteBastion is designed to support logistics and intelligent transportation use cases where clients need route optimization without coupling directly to a single vendor. The broker is intended to:

- Accept route optimization requests with waypoints and vehicles.
- Select or route requests to available VRP providers.
- Represent provider limits, features, access methods, and constraints in a local data model.
- Track optimization requests, runs, statuses, costs, vehicles, and waypoints.
- Manage customers and API keys for external systems.
- Support operational goals such as low latency, scalability, high availability, fault tolerance, and observability.

The broker README identifies Google Routes API as the first VRP provider integration target.

## Broker Service

The broker is the central runtime service. It is a Go application using:

- Gin for HTTP routing.
- pgx for PostgreSQL access.
- sqlc-generated query code.
- OpenTelemetry for tracing/exporter setup.
- Viper-based environment configuration.
- Google Maps/Routes dependencies for provider integration work.

The service entry point is `apps/broker/cmd/main.go`. Startup loads environment configuration, decodes the encryption key, initializes tracing, creates the HTTP server, and handles graceful shutdown on `SIGINT` or `SIGTERM`.

Current broker routes are registered in `apps/broker/pkg/server/http.go`:

- `GET /health`
- `GET /v1/optimizations/sync`
- `POST /v1/back-office/customers/` (PS.: Route being sunsetted)
- `DELETE /v1/back-office/customers/:customerId` (PS.: Route being sunsetted)
- `POST /v1/back-office/providers/` (PS.: Route being sunsetted)

The broker is organized around internal modules:

- `customer`: customer creation/deactivation and API key generation support.
- `provider`: provider registration and provider metadata persistence.
- `optimization`: optimization request handling. The current sync implementation is still a placeholder that returns an empty result after a short delay.
- `health`: database-backed health endpoint.
- `infra/db`: PostgreSQL provider, transaction manager, and generated sqlc database access.
- `infra/clients`: route optimization client abstractions and implementations, including fake and Google Cloud clients.

## Broker Data Model

The broker database schema models the main VRP broker concepts:

- Customers and API keys.
- Customer constraints such as budget, availability, performance, security, and feature requirements.
- Vehicles and cargo kinds.
- Optimization requests, runs, waypoints, and assigned vehicles.
- Providers, access methods, provider constraints, and provider features.

The schema is stored in `apps/broker/sql/schema.sql`, with migrations under `apps/broker/sql/migrations/` and development seed data under `apps/broker/sql/seeds/`.

## Admin API

The admin API is a NestJS application running on Fastify. It uses:

- NestJS modules and dependency injection.
- `@nestjs/config` for environment configuration.
- Drizzle ORM and PostgreSQL.
- Zod and `nestjs-zod` for validation and serialization.
- Swagger documentation in development mode.
- Vitest for unit and e2e test scripts.

The application module wires configuration, database infrastructure, global Zod validation/serialization, exception filtering, and the users module.

Current admin API behavior is centered on user management:

- `POST /users` creates a user after checking for duplicate email addresses.
- `DELETE /users/:id` deletes a user after validating that the user exists.

The admin API has a broader Drizzle schema that overlaps with broker administration concepts, including users, customers, API keys, providers, provider access methods, provider constraints, provider features, constraints, and vehicles.

## Admin UI

The admin UI is a Vue 3 application built with Vite. It uses:

- Vue Router.
- Pinia for state management.
- PrimeVue and PrimeIcons.
- Tailwind CSS.
- Ky for HTTP requests.
- Vitest and Vue Test Utils for frontend tests.

The UI is currently in an early state. `App.vue` only renders the router view, and the router defines route constants for authentication, password recovery, and dashboard paths, but no route records are registered yet.

## Local Development Infrastructure

The development environment is described in `docker-compose.yaml`. It defines:

- `admin-api` on port `3000`, with debugger port `9229`.
- `admin-database` PostgreSQL on host port `5432`.
- `admin-cache` Redis on host port `6379`.
- `broker` on port `8090`, with Delve debugger port `40000`.
- `broker-database` PostgreSQL on host port `5433`.
- `broker-cache` Redis on host port `6380`.
- `api-gateway` using Kong in database-less mode.
- `migrations` using `migrate/migrate` to apply broker database migrations.

Kong development configuration is mounted from `apps/broker/config/kong/dev/config.yaml`.

## Deployment and Testing Assets

Kubernetes manifests under `k8s/` describe a local broker deployment and service, including a Kind cluster configuration. The broker deployment currently declares two replicas and resource requests/limits.

Broker load testing assets live under `apps/broker/tests/load/` and include k6 and Locust scripts plus helper shell scripts. The repository also has `docker-compose.stress.yaml` for stress-oriented infrastructure.

## Current Implementation Notes

This project is still under active construction. The broker has the strongest domain framing and database model. The admin API and admin UI exist as application shells with initial user-management and frontend scaffolding work. Some README files are still framework starter documentation and should not be treated as authoritative project documentation.

Known current-state details:

- The broker optimization endpoint is not yet a full provider-backed optimization workflow.
- API key middleware is present but commented out in broker route registration.
- Provider update/delete handlers exist as commented-out broker code.
- The admin UI route constants exist, but the router has no registered pages yet.
- The Docker Compose setup defines separate admin and broker databases/caches.
